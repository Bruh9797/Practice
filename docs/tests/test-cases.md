# Тест-кейсы ThermoSelect

## Общие условия

- тестовый профиль: `test`, чистая H2 и миграции Flyway;
- демонстрационный профиль: `demo`, пользователи `demo/demo12345` и `admin/admin12345`;
- все изменяющие HTTP-запросы отправляются с актуальным CSRF-токеном;
- для негативных сценариев проверяется единый формат ошибки
  `{timestamp,status,error,code,message,details,path}`.

## Авторизация и безопасность

| ID | Сценарий | Действия | Ожидаемый результат |
|---|---|---|---|
| AUTH-01 | Регистрация | Получить CSRF, зарегистрировать нового пользователя | `201`, роль `USER`; пароль хранится как BCrypt-хеш |
| AUTH-02 | Дублирующий email | Повторить регистрацию с тем же email в другом регистре | `409`, код `DUPLICATE_USER` |
| AUTH-03 | Вход | Войти корректными данными с CSRF | `200`; ID сессии заменён; `/api/auth/me` возвращает пользователя |
| AUTH-04 | Ошибочный вход | Передать неверный пароль | `401`, код `AUTHENTICATION_FAILED`; сессия не авторизована |
| AUTH-05 | CSRF | Выполнить POST без токена и с неверным токеном | `403`, код `CSRF_INVALID` |
| AUTH-06 | Выход | Выполнить logout с CSRF | `204`; сессия недействительна; выдан новый CSRF-токен |
| AUTH-07 | Матрица ролей | Обратиться к USER- и ADMIN-методам без входа, как USER и как ADMIN | `401` без входа; USER API доступен обеим ролям; ADMIN API — только ADMIN |
| AUTH-08 | Отзыв сессии | Заблокировать или удалить вошедшего пользователя и повторить его API-запрос | Доступ отозван сразу, без ожидания окончания сессии |
| AUTH-09 | Смена роли | Снять у активного администратора роль ADMIN | Следующий ADMIN-запрос получает `403`; `/me` отражает новую роль |
| AUTH-10 | Самозащита администратора | Попытаться заблокировать, удалить или снять ADMIN у самого себя | `409`/`400` с предметным кодом; учётная запись не изменена |
| AUTH-11 | Cookie сессии | Проверить заголовок `Set-Cookie` | `HttpOnly`, `SameSite=Lax`, время простоя 30 минут |

## Каталог, поиск и сравнение

| ID | Сценарий | Действия | Ожидаемый результат |
|---|---|---|---|
| CAT-01 | Начальный каталог | Запустить demo-профиль, выполнить поиск без фильтров | В базе 42 опубликованные записи; пагинация стабильна |
| CAT-02 | Категориальные фильтры | Отфильтровать семейство, производителя, применение и материал | Возвращаются только записи, удовлетворяющие всем активным группам |
| CAT-03 | Числовые пределы | Задать требуемые площадь, расход, мощность, температуру и давление | Каждая запись строго покрывает заданные требования |
| CAT-04 | Неизвестное значение | Задать обязательный числовой фильтр для записи с `NULL` | Запись исключена; `excludedUnknownCount` увеличен |
| CAT-05 | Текстовый поиск | Искать по производителю, модели, серии и описанию | Релевантные совпадения первыми; причина совпадения показана |
| CAT-06 | Ранжирование | Активировать по одному и несколько критериев | Балл 0–100; веса только активных групп нормализованы |
| CAT-07 | Разрешение равенств | Создать записи с одинаковым баллом | Выше полнота, затем производитель и модель по алфавиту |
| CAT-08 | Пагинация | Запросить соседние страницы и край за диапазоном | Нет дублей; корректны `totalElements/totalPages`; край возвращает пустую страницу |
| CAT-09 | Детальная карточка | Открыть опубликованный slug | Общие поля, области, материалы, факты, пределы и источники согласованы |
| CAT-10 | Невидимые статусы | Запросить DRAFT/ARCHIVED через пользовательский API | `404`; записи не участвуют в поиске и сравнении |
| CAT-11 | Сравнение | Сравнить 2, 3 и 4 разных ID | Порядок входных ID сохранён; неизвестные значения явно обозначены |
| CAT-12 | Лимит сравнения | Передать 1, 5 и повторяющиеся ID | `400`, код валидации; допустимы только 2–4 уникальных ID |
| CAT-13 | Гранулярность | Сравнить EXACT_CONFIGURATION, STANDARD_MODEL и SERIES | Достоверность влияет на балл; для SERIES показана необходимость уточнения |

## Администрирование

| ID | Сценарий | Действия | Ожидаемый результат |
|---|---|---|---|
| ADM-01 | Создание записи | Создать валидный DRAFT со справочными связями | `201`; версия `0`; пользовательский поиск запись не показывает |
| ADM-02 | Редактирование | Обновить поля с текущей `version` | `200`; данные изменены; версия увеличена |
| ADM-03 | Optimistic locking | Дважды обновить запись с одной исходной версией | Второй запрос получает `409`, код `OPTIMISTIC_LOCK_CONFLICT` |
| ADM-04 | Публикация | Перевести заполненный DRAFT в PUBLISHED | Запись появляется в поиске и доступна по slug |
| ADM-05 | Архивирование | Выполнить DELETE или сменить статус на ARCHIVED | Физического удаления нет; запись исчезает из пользовательского API |
| ADM-06 | Производители | Создать, изменить и удалить неиспользуемого производителя | CRUD успешен; удаление используемого производителя отклонено конфликтом |
| ADM-07 | Пользователи | Сменить роль, заблокировать, разблокировать и удалить другого пользователя | Изменения сохранены и немедленно влияют на авторизацию |
| ADM-08 | Валидация | Передать пустую модель, неверный slug, отрицательные пределы или min > max | `400`, `VALIDATION_ERROR`, подробности привязаны к полям |

## Фронтенд и сквозные сценарии

| ID | Сценарий | Действия | Ожидаемый результат |
|---|---|---|---|
| UI-01 | Route guard | Открыть `/catalog`, `/compare` или `/admin` без входа | Переход на вход; исходный URL можно продолжить после авторизации |
| UI-02 | Состояния данных | Смоделировать loading, ошибку API и пустой результат | Для каждого состояния есть отдельное понятное представление |
| UI-03 | URL фильтров | Применить фильтры, обновить страницу и перейти назад/вперёд | Состояние фильтров восстанавливается из URL |
| UI-04 | Пояснение рейтинга | Открыть выдачу с несколькими критериями | Видны балл и причины: совпадения, запас, пробелы и предупреждение серии |
| UI-05 | Локальный лимит сравнения | Выбрать последовательно пять карточек | В список попадают четыре; пятая отклонена с сообщением |
| UI-06 | Административная форма | Создать/изменить запись, получить validation и version conflict | Успех и ошибки показаны без потери введённых данных |
| E2E-01 | Пользовательский путь | Регистрация → вход → поиск → карточка → сравнение | Весь сценарий проходит в одном JAR и одной origin |
| E2E-02 | Административный путь | Вход ADMIN → редактирование каталога → управление пользователем | Изменения видны в UI и применяются к доступу сразу |
| E2E-03 | Адаптивность | Повторить основные экраны на desktop и mobile | Фильтры пригодны на узком экране; сравнение прокручивается горизонтально |
| E2E-04 | SPA fallback | Напрямую открыть вложенные `/catalog/{slug}` и `/admin/catalog/{id}` | Сервер отдаёт React только UI-маршрутам; неизвестный `/api/**` остаётся API-ошибкой |

## Автоматизация проверки

- `mvnw.cmd clean verify` запускает unit/integration-тесты Java, `npm ci`, Vitest и production-сборку React.
- Браузерная smoke-проверка запускается против собранного JAR в профиле `demo`.
- Для PostgreSQL отдельно проверяются миграции Flyway и `spring.jpa.hibernate.ddl-auto=validate`.
