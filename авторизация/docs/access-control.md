# Авторизация и права доступа (модуль Тиграна)

## 1. Роли системы

| Роль  | Кто это | Возможности |
| --- | --- | --- |
| Гость (не авторизован) | Любой посетитель | Регистрация, вход, просмотр/поиск каталога теплообменных аппаратов (только чтение) |
| `USER` | Зарегистрированный пользователь | Всё, что доступно гостю, + просмотр и редактирование собственного профиля, сохранение истории/результатов поиска (при наличии такой функции у Сережи) |
| `ADMIN` | Администратор системы | Всё, что доступно `USER`, + управление каталогом теплообменных аппаратов (CRUD), управление пользователями и их ролями, просмотр журналов действий и отчетов |

Роль `USER` присваивается автоматически при самостоятельной регистрации.
Роль `ADMIN` не может быть получена через публичную регистрацию — она либо выдаётся
существующим администратором (`PATCH /api/admin/users/{id}/role`), либо принадлежит
служебной учетной записи, создаваемой автоматически при первом запуске системы
(см. `AdminAccountInitializer`, логин/пароль настраиваются через переменные окружения
`ADMIN_USERNAME` / `ADMIN_PASSWORD`).

## 2. Матрица доступа к API

| Endpoint | Гость | USER | ADMIN |
| --- | --- | --- | --- |
| `POST /api/auth/register` | ✅ | ✅ | ✅ |
| `POST /api/auth/login` | ✅ | ✅ | ✅ |
| `POST /api/auth/logout` | ❌ | ✅ | ✅ |
| `GET /api/auth/me` | ❌ | ✅ | ✅ |
| `GET /api/heat-exchangers/**`, `GET /api/search/**` (поиск, каталог) | ✅ | ✅ | ✅ |
| `POST/PUT/DELETE /api/heat-exchangers/**` (управление каталогом, модуль Сережи) | ❌ | ❌ | ✅ |
| `GET /api/admin/users` — список пользователей | ❌ | ❌ | ✅ |
| `PATCH /api/admin/users/{id}/role` — изменение роли | ❌ | ❌ | ✅ |
| `PATCH /api/admin/users/{id}/enabled` — блокировка/разблокировка | ❌ | ❌ | ✅ |
| `DELETE /api/admin/users/{id}` — удаление пользователя | ❌ | ❌ | ✅ |
| `GET /api/admin/logs`, `GET /api/admin/reports` (модуль Милана) | ❌ | ❌ | ✅ |

Правило `/api/admin/**` → `hasRole("ADMIN")` задано на уровне `SecurityFilterChain`
и дополнительно продублировано аннотацией `@PreAuthorize("hasRole('ADMIN')")`
на контроллере (принцип defense-in-depth: даже если правило маршрута случайно
изменят, метод останется защищённым).

Администратор не может понизить в правах, заблокировать или удалить
собственную учетную запись — это отдельно проверяется в `AdminUserController`,
чтобы не оставить систему без единственного администратора.

## 3. Технические решения

- Хранение пароля: **BCrypt** (`BCryptPasswordEncoder`), пароли в открытом виде нигде не хранятся и не логируются.
- Механизм входа: сессионная аутентификация Spring Security (`HttpSession` + `SecurityContext`),
  без CSRF-токенов, т.к. фронтенд работает как отдельное SPA-приложение через JSON API
  (в проде рекомендуется настроить `SameSite`/`Secure` cookie и сузить CORS-origins).
- Ошибки авторизации возвращаются в едином JSON-формате:
  - `401 Unauthorized` — не выполнен вход (`JsonAuthenticationEntryPoint`);
  - `403 Forbidden` — выполнен вход, но роли не хватает (`JsonAccessDeniedHandler`);
  - `409 Conflict` — конфликт данных (например, логин уже занят, попытка удалить себя).

## 4. Сценарии для UML-диаграмм вариантов использования (Use Case)

### Актор: Пользователь (Гость / USER)

1. **Регистрация** — вводит логин, email, пароль → система создаёт учетную запись с ролью `USER`.
2. **Вход в систему** — вводит логин/пароль → система проверяет учетные данные и открывает сессию.
3. **Поиск теплообменных аппаратов** — задаёт параметры фильтрации → получает список/карточки результатов
   (доступно и без входа, для авторизованных — с сохранением истории, если реализовано).
4. **Просмотр карточки аппарата** — открывает детальную информацию по одному аппарату.
5. **Просмотр собственного профиля** — `GET /api/auth/me`.
6. **Выход из системы** — завершает сессию.

### Актор: Администратор (ADMIN)

Наследует все варианты использования актора "Пользователь" (отношение `«extend»`/generalization
на UML-диаграмме) и дополнительно:

1. **Управление каталогом аппаратов** — добавление, редактирование, удаление записей теплообменников.
2. **Управление пользователями** — просмотр списка пользователей, изменение роли, блокировка/разблокировка,
   удаление учетной записи.
3. **Просмотр журналов действий** — анализ логов действий пользователей и ошибок (модуль Милана).
4. **Просмотр отчетов** — выгрузка отчетных форм по работе системы (модуль Милана).
